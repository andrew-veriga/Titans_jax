# План миграции проекта Gemma-Titans на фреймворк Kauldron

## Введение
Данный документ описывает шаги по переводу прототипа модели Gemma-Titans (с интегрированной нейронной памятью) с ручного цикла обучения (чистый JAX + Optax) на высокоуровневый фреймворк `Kauldron`. 

Переход на Kauldron необходим для масштабирования обучения на мульти-TPU кластеры (TPU Pods), использования FSDP (Fully Sharded Data Parallelism) и продвинутого профилирования памяти/ресурсов.

## Этап 1: Подготовка архитектуры модели
1. **Рефакторинг возвращаемых типов:**
   - В текущем MVP `GemmaTitansTransformer` возвращает нестандартный `TitansLayerCache`. Kauldron имеет строгую систему проверки типов (через `kauldron.typing`).
   - *Действие:* Адаптировать `TitansLayerCache` к стандартам `kd.nn` или написать корректные type hints, чтобы избежать необходимости отключать проверки (`KAULDRON_TYPECHECK='0'`).
2. **Обертка модели (Model Wrapper):**
   - Упаковать `GemmaTitansTransformer` в структуру, совместимую с `kd.nn.Module` (или удостовериться, что текущая реализация Flax Linen корректно подхватывается Kauldron).

## Этап 2: Инициализация и загрузка весов (InitTransform)
1. **Реализация `SkipTitans`:**
   - Написать класс `SkipTitans`, наследующий `kd.ckpts.AbstractPartialLoader`.
   - Внедрить логику из `Kauldron_SkipTitans_Research.md`: разделение дерева параметров на базовые (Gemma) и новые (Titans: `memory`, `memory_gate`).
   - Обеспечить корректное слияние загруженных весов (с Kaggle/HuggingFace) и случайно инициализированных модулей памяти.
2. **Конфигурация инициализатора:**
   - Настроить `init_transform` для передачи в `Trainer`:
     ```python
     init_transform = SkipTitans(
         wrapped=gm.ckpts.LoadCheckpoint(path=CKPT_PATH)
     )
     ```

## Этап 3: Пайплайн данных (Data Pipeline)
1. **Переход на Grain / SeqIO:**
   - Заменить ручную загрузку `datasets` на `kd.data` (в связке с `grain` или `seqio`).
   - Настроить асинхронные итераторы для обеспечения максимальной пропускной способности (throughput) на TPU.
   - Написать логику токенизации внутри графа пайплайна данных.

## Этап 4: Конфигурация Kauldron Trainer
1. **Создание файла конфигурации (Config / ConfigDict):**
   - Определить гиперпараметры обучения, лосс-функцию, оптимизатор (Optax) внутри декларативного конфига Kauldron.
2. **Настройка стратегии шардирования (Sharding):**
   - Настроить `kd.sharding.ShardingStrategy` для распределения весов.
   - Для 1 TPU v5e: использовать простую репликацию или базовый data-parallel.
   - Для TPU Pod (например, v5e-8): настроить оси FSDP/Tensor Parallel для распределения `embed_dim` и модулей памяти между чипами.
3. **Инициализация тренера:**
   - Собрать `kd.train.Trainer` со всеми компонентами (model, init_transform, data, optimizer, sharding).

## Этап 5: Профилирование и запуск
1. **AOT Компиляция и Анализ памяти:**
   - Использовать встроенные средства JAX/Kauldron для предварительной компиляции шага обучения (`Trainer.train_step`) и оценки потребления HBM (памяти ускорителя) перед запуском.
2. **Интеграция с TensorBoard:**
   - Настроить логирование метрик лосса, градиентов (полезно для отладки взрывающихся градиентов в модулях памяти) и профилей XLA.
3. **Запуск обучения (CPT - Continued Pre-Training):**
   - Запустить процесс файнтюнинга модулей Titans на целевом датасете (например, OpenAssistant).

## Риски и сложности (Mitigation)
- **Строгая типизация Kauldron:** Может потребовать глубокого вмешательства в аннотации типов `gemma.gm.nn`.
- **Нестандартный Cache:** При генерации текста (inference) интеграция `TitansLayerCache` в стандартный цикл `kd.train.Evaluator` может потребовать написания кастомного шага оценки.
